# Replit Agent Task: Full Codebase & Documentation Audit

## Objective

Perform a comprehensive audit of this project. First, identify and triage dead code that was previously functional but got broken or bypassed by prior agent sessions — present a restoration plan and wait for approval before fixing. Then audit the full codebase, compare against existing documentation, identify drift, and generate senior-grade documentation that reflects the true current state. Do not guess. Read every file.

## Phase 0 — Dead Code Restoration (RUN FIRST)

**Context:** Previous agent sessions broke working functionality by rewriting or replacing code, then left the original working versions in the codebase as "dead code" — commented out, renamed with `.old`/`.bak`/`.backup` suffixes, moved to unused files, or bypassed by new imports. That "dead code" was functional. I want it back.

### 0.1 Identify All Candidates

Scan the entire codebase for code that was likely once active but got sidelined:

- **Commented-out blocks** — Functions, components, routes, middleware, or logic blocks that are commented out but structurally complete. Look for `//`, `/* */`, `{/* */}`, `#`, `"""`, `<!-- -->` wrapping what looks like working code.
- **Renamed/suffixed files** — Files ending in `.old`, `.bak`, `.backup`, `.orig`, `.deprecated`, `.unused`, `_old`, `_backup`, `_v1`, `_prev`, or similar.
- **Duplicate implementations** — Two versions of the same function/component/route where one is imported and the other isn't. The unused one may be the original.
- **Orphaned exports** — Functions/components/classes that are exported but never imported anywhere. These may have been the active version before a rewrite.
- **Git history clues** — Run `git log --oneline -20` and `git diff HEAD~5 --stat` to identify recently changed files. These are the most likely regression sites.
- **TODO/FIXME markers referencing old behavior** — Comments like `// TODO: restore old logic`, `// was working before`, `// reverted`, `// old version below`.
- **Conditional dead branches** — `if (false)` blocks, feature flags set to permanently off, or environment checks that can never be true.

### 0.2 Triage Each Candidate

For EVERY dead code candidate found, produce a triage entry:

```
### [File Path]:[Line Range]

**Type:** Commented-out block | Renamed file | Orphaned export | Duplicate implementation | Dead branch
**What it does:** [1-2 sentence description of the dead code's functionality]
**Current replacement:** [What code is currently doing this job instead, if any — file path + line]
**Current replacement working?** YES | NO | PARTIALLY | UNTESTED
**Recommendation:** RESTORE | KEEP_CURRENT | MERGE | NEEDS_DECISION

**If RESTORE:** [Explain what to uncomment/rename/re-import and what to remove]
**If MERGE:** [Explain what to take from each version]
**If NEEDS_DECISION:** [Explain the tradeoff — present both versions with what each does differently]
```

### 0.3 Restoration Rules

1. **Do NOT delete any dead code yet.** Triage everything first. Present the full report before making changes.
2. **Do NOT assume the newer version is better.** The older "dead" version may have been working correctly before the agent rewrote it.
3. **Test both versions** where possible — if there are tests, run them against both the current and dead versions to see which passes.
4. **Check git blame** on the dead code vs the replacement — if the dead code is from an earlier human commit and the replacement is from a recent agent commit, bias toward restoring the dead code.
5. **Preserve the replacement** as a comment or `.backup` file when restoring, so nothing is permanently lost.
6. **After triage is complete**, present the full report and WAIT for my approval before making any restorations. Do not auto-restore.

### 0.4 Output: DEAD-CODE-TRIAGE.md

Generate `/docs/DEAD-CODE-TRIAGE.md` containing:
- Total candidates found (count)
- Breakdown by type (commented-out, renamed files, orphaned exports, etc.)
- Each triage entry from 0.2 above
- Summary table:

```
| # | File | Type | Recommendation | Risk if Restored |
|---|------|------|----------------|-----------------|
| 1 | src/auth/middleware.js:45-89 | Commented-out | RESTORE | Low — tests pass |
| 2 | components/Dashboard.old.tsx | Renamed file | NEEDS_DECISION | Medium — UI changed |
...
```

- At the bottom: "Awaiting approval to proceed with restorations."

**STOP after Phase 0 and present DEAD-CODE-TRIAGE.md before continuing to Phase 1.** I will review and approve/modify the restoration plan. Then you execute the approved restorations. Then proceed to Phase 1.

## Phase 1 — Codebase Audit

Scan the entire repository. For each of the following, document what actually exists (not what should exist):

### 1.1 Project Structure
- Map the full directory tree (exclude `node_modules`, `.git`, build artifacts)
- Identify the framework, language, and runtime (versions from `package.json`, `requirements.txt`, `.python-version`, `replit.nix`, or equivalent)
- List all entry points (server start, build commands, scripts)
- Identify environment configuration files (`.env.example`, `.replit`, `replit.nix`, `vercel.json`, `wrangler.toml`, etc.)

### 1.2 Architecture
- Map all routes/endpoints (REST, GraphQL, WebSocket — whatever exists)
- Identify the data layer (database, ORM, direct queries, migrations)
- Map authentication and authorization flows (auth provider, session handling, middleware)
- Document all external service integrations (APIs, SDKs, webhooks)
- Identify background jobs, cron tasks, or queue workers if present
- Map state management approach (client-side and server-side)

### 1.3 Dependencies
- List all production dependencies with versions
- List all dev dependencies with versions
- Flag any dependencies with known vulnerabilities (`npm audit` or equivalent)
- Flag any outdated major versions (compare against latest stable)
- Identify unused dependencies (imported but never referenced in code)

### 1.4 Security Posture
- Check for hardcoded secrets, API keys, tokens, or credentials in any file (including comments and dead code)
- Verify environment variables are used for all sensitive values
- Check for input validation on external boundaries (API routes, form handlers, URL params)
- Check for SQL injection vectors (raw queries without parameterization)
- Check for XSS vectors (unsanitized output rendering)
- Check authentication middleware coverage (which routes are protected, which are open)
- Check CORS configuration
- Check rate limiting presence
- Check security headers (HSTS, CSP, X-Frame-Options, X-Content-Type-Options)
- If using Supabase: verify Row-Level Security (RLS) is enabled on all tables containing user data

### 1.5 Testing
- Identify test framework and configuration
- Count total tests, passing, failing, skipped
- Calculate approximate code coverage (or note if coverage tooling is missing)
- Identify untested critical paths (auth, payments, data mutations)
- Run the test suite and capture results

### 1.6 Code Quality
- Identify linting configuration (ESLint, Prettier, Ruff, etc.) — present or missing
- Check TypeScript strict mode (if TypeScript project)
- Flag any `any` types, `@ts-ignore`, or `eslint-disable` overrides
- Identify dead code (exported but never imported, unreachable branches) — cross-reference with Phase 0 triage. Any dead code already addressed in Phase 0 restorations should be marked as resolved, not re-flagged.
- Check error handling patterns (try/catch coverage, error boundaries, graceful degradation)
- Check logging patterns (structured vs console.log, log levels)
